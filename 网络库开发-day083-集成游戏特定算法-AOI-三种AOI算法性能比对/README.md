# Day83： 集成游戏特定算法  性能比对测试

## 性能比对

-- 1.  在进行5000个不同的玩家登录地图， 10000次的玩家在地图里面移动， 5000次的查询视野内的玩家。同时地图设置长宽为 (1024 乘以 1024）。 视野范围为(150 乘以 150). 得到下面的性能分析数据。
  
 |算法	|插入 (ms)	|移动 (ms)	|查询 (ms)	|总耗时 (ms)	|综合评价|
|-------|-------------|-------------|-------------|-------------|---------|
|四叉树	|0.80	|32.51	|3.04	|36.36	|🏆 最优|
|网格法	|4.64	|66.09	|23.98	|94.70	|✅ 优秀|
|十字链表	|505.61	|4536.48	|2.63	|5044.72	|❌ 最差|


## 数据解读
###  为什么十字链表这么差？

-- 1. 十字链表的优势在于查询（O(1) 从当前节点向四个方向遍历），但它的插入和移动是O(n) 的链表遍历。

-- 2. 在 5000 个实体规模下，每次插入或移动都需要遍历平均 2500 个节点来找到正确的插入位置，累积起来就是 4500 毫秒的移动耗时。这意味着在实体数量超过几百的场景下，十字链表就已经不适合了。

-- 3. 它适用于“实体数量少、查询频繁”的逻辑密集型场景，比如回合制游戏或策略游戏，而在你现在的测试条件（5000 实体、大量移动）下，它的弱点被放大了。


###  为什么四叉树最优？

-- 1.  四叉树的插入和移动是 O(log n) 的树深度（深度最多 8 层），5000 个实体下每次操作最多 8 次比较，所以插入 0.8ms、移动 32ms 是完全合理的。

-- 2. 它的查询也很高效，因为 Query 只收集候选区域内的实体，然后做网格过滤。

-- 3.  四叉树在你这个测试条件下综合表现最好。

### 网格法的表现很稳健

-- 1. 网格法的插入、移动、查询都是 O(1) 的取模运算，没有复杂的数据结构维护。

-- 2.  移动 66ms 是因为每次移动都要更新两个网格的 set，删除和插入各一次，但整体仍然非常快。

-- 3.  网格法的查询是九宫格范围的固定扫描，23ms 属于正常范围。


##  结论与建议

-- 1. 四叉树综合最优：如果你的场景是“实体数量大、移动频繁、分布不均匀”，四叉树是首选。

-- 2.  网格法稳定可靠：如果实现简单、维护成本低是优先考虑因素，网格法仍然是不错的选择。

-- 3.  十字链表需要谨慎使用：只有在“实体数量少、查询极频繁、移动极少”的场景下（比如回合制战斗中的 AOI 遍历），十字链表才有优势。
