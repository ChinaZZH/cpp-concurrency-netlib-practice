# Day81： 集成游戏特定算法  十字链表法

## 核心收获

### 九宫格法和其他AOI算法的对比 

|算法|	核心特点|	最佳适用场景|	慎用场景|
|-----|-----------|---------------|-----------|
|网格法（九宫格）|	简单直接，按位置划分固定格子	| 平坦开阔地图、玩家分布相对均匀（如 MOBA、大逃杀）|	玩家高度聚集（如主城摆摊）、地图地形复杂多障碍|
|十字链表|	基于双向链表维护 X/Y 轴顺序，擅长遍历|	逻辑密集型场景、需要快速 AOI 遍历（如回合制游戏、策略游戏）|	高频创建/删除实体（如频繁进出副本）|
|四叉树|	自适应分区，根据对象密度动态调整	| 地形复杂、对象分布不均匀（如开放世界 RPG、场景中有大量障碍物）|	需要极高性能的实时竞技游戏（因为动态分区和移动更新开销较大）|


###  统一基类 BaseAOIManager 与接口 IAOIManager
-- 1. 抽象出了 IAOIManager 接口，定义了所有 AOI 算法必须实现的公开方法。

-- 2. 同时设计了 BaseAOIManager 作为公共基类，把网格坐标计算、广播回调注入等通用功能抽取到基类中，让网格法、十字链表、四叉树三种算法能够无缝切换。

###  消息通知器 AOIMsgNotifyer 的抽离

-- 1. Protobuf 消息构造和广播发送逻辑从 AOIManager 中剥离，独立为 AOIMsgNotifyer。

-- 2. 事件发生时，算法只调用 msgNotifyer_ 的接口，由它负责构造 EntityEnterNotifyResponse、InitAroundEntitiesNotifyResponse 等消息并广播。

-- 3. 这样算法实现不再依赖消息格式，解耦更彻底，也为后续单元测试提供了便利。

### 网格坐标缓存与单向链表遍历

-- 1.  十字链表节点中缓存了 gridX 和 gridY，避免每次 GetNeighbors 都重新计算网格坐标。

-- 2. 在 GetNeighbors 中，你从当前节点沿 X 轴和 Y 轴各向两个方向遍历，收集候选集合后取交集，并利用 std::set 自动去重。

## 结论

--  1. 新增, 移动，都是需要从headX和headY 从头到尾的遍历，需要消耗 o(n)性能

--  2.  查找性能和视野内同步较快，因为从当前节点往x轴和y轴链表扩散

--  3.  主要针对实体数量少， 查询和视野内同步高频的场景使用。
