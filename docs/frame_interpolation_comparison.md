# 帧同步插值方案对比：帧号 vs 时间戳

## 背景

-- 1.  在帧同步架构中，客户端收到服务端广播的远程玩家位置更新时，不能直接跳变（否则画面会“瞬移”），需要在两次更新之间进行平滑插值。

-- 2.  插值需要两个已知状态：prev（较早的状态）和 next（较新的状态），以及一个决定插值进度的锚点。这个锚点的选择，直接影响系统的确定性、平滑度和工程复杂度。

-- 3.  本文对比两种主流方案：

方案 A：基于服务端帧号（server_frame）

方案 B：基于客户端本地时间戳（now_ms）


## 方案 A：基于服务端帧号

###  数据结构
```cpp
     struct RemoteStateSnapshot {
    Fixed x, y;
    uint32_t server_frame = 0;    // 服务端帧号（20ms 递增）
    bool valid = false;
};
```

###  插值流程

```text
    客户端收到广播包 → 存储 prev/next → 计算 render_frame = latest_server_frame - 2（落后 2 帧）
→ 在 prev 和 next 之间按帧号比例插值 → 渲染
```

###  优点

-- 1.  完全确定性：帧号由服务端统一生成，跨平台、跨运行完全一致。

-- 2.  无需时钟同步：不依赖客户端与服务器的系统时钟对齐。

-- 3.  断线重连自动恢复：重新连接后，只要有新的 server_frame，插值立即恢复。

-- 4.  与延迟补偿共享时间轴：攻击判定也使用 server_frame，整个系统时间锚点统一。

###  缺点

-- 1.  平滑度有限：帧号是离散整数，插值进度 t 取值为 offset/range，当广播间隔较大（如 50ms）时，可能出现微小阶梯感。

-- 2.  需要设置固定滞后帧数：RENDER_DELAY_FRAMES 是硬编码常量（如 2），无法动态适应网络波动。


## 方案 B：基于时间戳

###  数据结构
```cpp
struct RemoteStateSnapshot {
    Fixed x, y;
    uint64_t timestamp_ms = 0;   // 服务端生成时间戳
    bool valid = false;
};

```

###  插值流程

```text
客户端收到广播包 → 存储 prev/next → 计算 render_time = now_ms - LAG_MS（滞后固定毫秒数）
→ 在 prev 和 next 之间按时间比例插值 → 渲染
```

###  优点

-- 1.  理论上更平滑：时间戳是连续变化的，t 值可以无限细分，不会出现阶梯感.

-- 2.  可自适应调整：滞后量可以动态计算（如基于 RTT），适应不同网络环境。


###  缺点

-- 1.  依赖时钟同步：若客户端与服务器时钟偏差超过滞后量，插值窗口会偏移甚至失效。

-- 2.  断线重连需要重建基线：重连后本地时间与服务器时间的关系可能变化，需重新计算。

-- 3.  与延迟补偿两套时间轴：延迟补偿用 server_frame，插值用 ms，增加系统复杂度。

-- 4.  滞后量难以精确配置：固定滞后量过大（增加延迟）或过小（插值窗口不足）都会影响体验。


## 对比总览

|对比维度|			帧号方案|				时间戳方案|
|---------|-----------------|--------------------|
|平滑度|				有限（离散帧号）|	理论最优（连续时间）|
|确定性|				✅ 完全确定		   |❌ 依赖时钟同步，运行结果不可复现|
|时钟依赖	|			无							| 强依赖（客户端与服务器时钟需对齐）|
|断线重连|			自动恢复				| 需重建时间基线 |
|与延迟补偿对齐|	✅ 统一 server_frame |	❌ 两套时间轴|
|工程复杂度	|		低	 | 中（需处理 RTT 估算、时钟漂移）|
|配置参数	|			RENDER_DELAY_FRAMES（固定）|	LAG_MS（固定）或动态计算 |


##  为什么不选“理论上更平滑”的时间戳方案？

-- 1.   因为在帧同步架构中，确定性 > 平滑度。

-- 2.   帧同步的核心假设是：所有参与方在相同输入下，必须得出相同的结果。

-- 3   如果插值结果依赖于客户端本地的时钟读数（now_ms），而不同客户端时钟不同步，则：

-- 4.   同一帧在不同设备上渲染的位置可能不同（虽然差异微小）。

-- 5.   回放录像时，由于时间轴无法复现，插值结果与原始运行不同，导致录像失效。

-- 6.   断线重连后，客户端无法精确恢复到断线前的位置。

-- 7.   因此，即便时间戳方案在理论上更平滑，帧同步引擎普遍选择帧号方案——牺牲一点平滑度，换取系统的确定性和跨平台一致性。

-- 8.   这是帧同步与状态同步的核心区别之一：帧同步优先保障确定性，状态同步优先保障表现平滑


##  混合方案（状态同步常用）

-- 1.   在状态同步架构中（服务器下发权威位置，客户端做插值），常用混合方案：

-- 2.   服务器下发 (position, timestamp)

-- 3.   客户端计算 render_time = now_ms - RTT/2，用时间戳做插值

-- 4.   原因：状态同步不要求确定性（服务器是唯一权威），可以优先保障视觉平滑。

-- 5.   这种方案在 FPS、MOBA 的状态同步实现中非常常见，但它牺牲了回放一致性和跨平台确定性，代价不同。

##   主流帧同步游戏服务器用哪个？

-- 1.  结论：绝大多数采用“帧号方案”，或者以帧号为核心、时间戳为辅助的混合方案。

-- 2.    确定性是铁律：帧同步要求所有客户端在相同输入下产生完全一致的结果。时间戳方案对时钟同步和网络抖动敏感，可能导致不同客户端在同一帧产生微小差异，破坏确定性。

-- 3.   工业标准（GGPO）：格斗游戏领域的黄金标准 GGPO（Good Game Peace Out），就是基于帧号（Frame） 来工作的。

```text
商业引擎与框架实践：

Unity 的帧同步框架强调“使用固定的时间片段作为阀值”，即帧号方案。

Unreal Engine 的网络时间系统同时使用 frameId 和 timeMs，体现了以帧号为主的混合思路。

商业项目 MultiVersus 使用Rollback Netcode时，每个按键都附带“精确时间戳”。但这是在确定性的帧号框架内，为延迟补偿等特定功能增加辅助精度。

为什么“以帧为主，时间为辅”是主流？
根本原因（确定性）：帧号是确定性游戏世界的“心跳”。用帧号做插值，保证了这个心跳的纯净，维护了整个系统的确定性根基。

辅助手段（平滑度）：在确定了“第103帧”这个核心锚点后，引入时间戳或固定时间步长（如16.67ms）来做渲染插值，是为了在保持确定性的前提下，让画面更平滑。
```

##  结论
|场景		| 推荐方案			|
|---------|-----------------|-
|帧同步（需要确定性回放、跨平台一致）	 |帧号方案 |
|状态同步（服务器绝对权威，优先视觉）	 |时间戳方案（或混合方案）|
|帧同步 + 需要高平滑度（如赛车游戏）	 |帧号方案 + 客户端额外平滑滤波（如指数平滑）|

本次训练中采用帧号方案，与定点数、延迟补偿、服务端矫正统一在同一时间轴下，是帧同步架构的正确选择。
