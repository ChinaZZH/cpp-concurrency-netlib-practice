## 网络库性能测试报告
## 测试环境
CPU：12 核（Intel / AMD）

内存：16 GB

操作系统：Ubuntu 22.04（WSL 2）

编译器：g++ 11.4，-std=c++17 -O2 -pthread

压测工具：wrk 4 线程，1000 并发，持续 30 秒

## 测试配置
|配置项|	值|
|------|----|
|服务器处理|	HTTP Hello, World!（同步回显）|
|线程池|	EventLoopThreadPool（工作线程）|
|主线程|	独立 EventLoop，只负责 accept|
|任务线程池|	未使用（业务逻辑在 IO 线程直接处理）|

## 性能分析
从单线程到多线程：QPS 从 40k 提升至 77k，提升幅度约 94%，平均延迟降低 50%，P99 延迟降低 30%，超时完全消除。

最佳工作线程数：在 12 核机器上，4~6 个工作线程达到性能峰值；继续增加线程（8）反而引起轻微下降，可能是锁竞争或调度开销增加所致。

CPU 利用率：多线程版本成功将负载分散到多个核心，4 线程时 CPU 总占用约 429%，6 线程时约 439%，说明主要瓶颈不在 CPU，可能已接近网络栈或 wrk 客户端的极限。

## 结论
本网络库在多核环境下具有良好的可扩展性，能够充分利用多核 CPU，在普通开发机上达到 77k QPS，且延迟稳定。该性能足以满足绝大多数后端服务的需求。

## 后续优化方向

使用 io_uring 替换 epoll 进一步提升 I/O 效率

针对静态文件服务使用 sendfile 零拷贝

将连接管理下放到工作线程（消除主线程 map 锁）
