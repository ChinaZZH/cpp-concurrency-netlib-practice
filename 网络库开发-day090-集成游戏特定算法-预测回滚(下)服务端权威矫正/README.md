# Day89： 集成游戏特定算法  预测回滚(上) -客户端预测+回滚重放

## 核心收获

-- 1.  客户端预测+回滚重放 本质上就是数据库时间点恢复在游戏领域的标准映射。
 
-- 2.  客户端每20毫秒执行一个Tick 帧号自增 (current_client_frame_ += 1)  并且拿到当前帧号的操作记录然后进行模拟预测，最后保存当前当前时刻的一个全量快照。
 
-- 3.  客户端每一个操作OnLocalInput 都将当前的操作保存在未来帧操作记录里面里面。未来帧帧号  (uint32_t predicted_frame = current_client_frame_ + 1). 

-- 4.  当服务端回复的客户端帧号远远小于客户端当前的帧号。则通过回复的客户端帧号取得当前的快照，当前数据回到这个全量快照的时刻。然后把大于这个时刻的操作记录执行一遍(重放)。 这个就是回滚重放。

-- 5.  这个原理类似于mysql的主从同步。

## 数据库时间点恢复的比较
|数据库原理 |	客户端预测实现+回滚重放|	作用|
|------------|-----------------------------|-------|
|全量备份（Full Backup）|	snapshots_.Save() 保存的 PlayerSnapshot	| 某一帧时刻的完整世界状态（位置、速度、血量）|
|增量日志（Binlog / Redo Log）|	pending_inputs_（std::map 中的输入帧）	| 按时间顺序（帧号）排列的所有操作记录|
|恢复工具（mysqlbinlog）|	Rollback(target_frame) + Replay()	 | 从全量备份出发，按顺序重放日志，将状态“快进”到最新时刻|
|日志清理（Purge）|	OnServerFrame 中 erase(<= acked_frame) |	确认该帧已被服务端归档（落盘/广播），本地可以安全删除|
|主从同步校验	| “服务端权威矫正”	| 主库（服务端）算出权威结果，对比从库（客户端）的查询结果是否一致|
