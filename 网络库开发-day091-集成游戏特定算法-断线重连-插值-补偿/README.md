# Day91： 集成游戏特定算法  体验优化( 断线重连，平滑插值， 延迟补偿)

##  概述

-- 1.   **断线重连**：网络中断后，玩家能恢复到断线前的状态，而非重新开始。

-- 2.   **平滑插值**：消除服务端广播频率（50ms）带来的其他玩家移动“跳帧”感。

-- 3.  **延迟补偿**：解决“我瞄准了，但子弹判定没打中”的网络延迟偏差问题。


##   核心设计决策：统一的服务端帧号体系

-- 1.   **关键决策**：（断线重连、平滑插值、延迟补偿）全部统一使用**服务端逻辑帧号（`server_frame`，20ms 递增）**作为时间轴锚点。

-- 2.  **决策背景**：

- 初始设计中，平滑插值使用了客户端本地时间戳（`now_ms`）作为插值基准，但客户端与服务器时钟不同步，导致插值窗口不稳定。

- 延迟补偿最初使用客户端帧号（`client_frame`）查询历史，但客户端帧号在断线重连后可能重置，且空输入帧不存储历史，导致查询失败。

-- 3.  **决策收益**：

- 消除客户端-服务端时钟同步依赖。

- 服务端历史缓存每 20ms 无条件存储，按 `server_frame` 查询永远能命中。

- 所有补偿和回溯操作基于同一套确定性帧号体系。


## 功能一：断线重连（Reconnection）

###  解决的问题
客户端因网络波动或进程重启导致连接中断后，重新连接时无法恢复断线前的状态（位置、血量等），需要从零开始。

### 核心设计

| 角色 | 职责 |
| :--- | :--- |
| 客户端 | 检测到断线后，发送 `ReconnectRequest` 请求全量快照。 |
| 服务端 | 收到请求后，从 `ServerPlayerManager` 中提取当前权威状态，打包为 `SnapshotReply` 回复。 |
| 客户端 | 收到 `SnapshotReply` 后，覆盖本地状态，清空历史快照和 `pending_inputs_`，继续模拟。 |



## 功能二：平滑插值（Smooth Interpolation）

###  解决的问题

-  服务端广播间隔为 50ms（20 FPS），若客户端直接使用最新位置渲染，其他玩家会产生明显的“跳帧”感，影响竞技体验。

### 核心设计

-  双缓冲：为每个远程玩家维护 prev_ 和 next_ 两个状态，分别对应最近两次收到的广播帧。

-  滞后渲染：客户端不追最新帧，而是落后 2 帧（约 40ms）进行渲染，让渲染帧号始终落在 prev_ 和 next_ 之间。

-  线性插值：在 prev_ 和 next_ 之间按帧号比例计算中间位置，实现平滑过渡。


## 功能三：延迟补偿（Lag Compensation）

###  解决的问题

-  客户端开火时，目标在屏幕上位于位置 A。由于网络延迟，攻击包到达服务器时，目标可能已移动到位置 B。若服务器用实时位置 B 进行判定，玩家会感觉“明明瞄准了却打不中”。

### 核心设计

-  服务端历史缓存：ServerPlayerManager 为每个玩家维护一个环形缓冲区（容量 64 帧，约 1.28 秒），每 20ms 存储一次状态。

-  攻击携带帧号：客户端发送 AttackRequest 时，携带当前收到的最新服务端帧号（latest_server_frame_）。

-  回溯判定：服务端收到攻击请求后，用 server_frame 查询目标在“开枪那一刻”的历史位置，用历史位置做碰撞检测。
